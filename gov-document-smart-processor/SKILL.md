# 政务公文智能处理助手

## 触发条件

当用户出现以下任意场景时，使用本 Skill：
- 需要起草、修改或校验党政机关公文
- 提到"公文"、"通知"、"请示"、"函"、"报告"、"纪要"
- 需要校对公文格式（标题、称谓、落款、编号等）
- 需要从政策文件中提取要点或做摘要
- 需要对比分析多份政策文件
- 需要生成会议纪要或信息简报
- 需要对公众留言进行分类和生成答复

## 核心能力

1. **公文智能起草**：按文种模板生成符合规范的公文初稿
2. **格式自动校验**：依据国标 GB/T 9704 检查公文格式要素
3. **关键信息提取**：从公文中自动提取发文单位、主题、关键词、时限要求等
4. **政策摘要与对比**：多份政策文件的结构化摘要和差异对比
5. **会议纪要生成**：从会议记录中提炼结构化纪要
6. **公众留言处理**：分类、判定紧急程度、生成标准化答复

## 操作步骤

### 步骤 1：识别任务类型

根据用户输入判断任务类型：

| 任务类型 | 触发词 | 对应步骤 |
|---|---|---|
| 公文起草 | "写/起草/拟一份+文种" | 步骤 2-4 |
| 格式校验 | "检查/校验/核对+格式" | 步骤 5 |
| 信息提取 | "提取/摘要/归纳+公文/政策" | 步骤 6 |
| 政策对比 | "对比/比较+政策/文件" | 步骤 7 |
| 会议纪要 | "纪要/会议记录/简报" | 步骤 8 |
| 留言处理 | "留言/咨询/投诉+分类/回复" | 步骤 9 |

### 步骤 2：公文起草 - 确认要素

收集公文起草必需信息：

| 要素 | 必填 | 说明 |
|---|---|---|
| 文种 | 是 | 通知/请示/报告/函/批复/意见/通报/决定 |
| 发文机关 | 是 | 全称或规范化简称 |
| 主送机关 | 是 | 接收单位全称 |
| 主题/事由 | 是 | 公文核心内容 |
| 正文要点 | 是 | 需要传达的关键信息 |
| 附件（如有） | 否 | 附件名称和份数 |
| 紧急程度 | 否 | 特急/加急/平件（默认平件） |
| 密级 | 否 | 绝密/机密/秘密/公开（默认公开） |

### 步骤 3：加载文种模板

读取 `templates/doc-{文种}.md` 获取对应文种的结构模板和格式规范。

**支持文种**：
- `notice`（通知）—— 要求下级办理或知晓事项
- `request`（请示）—— 向上级请求指示或批准
- `report`（报告）—— 向上级汇报工作、反映情况
- `letter`（函）—— 不相隶属机关之间商洽工作
- `reply`（批复）—— 答复下级请示
- `opinion`（意见）—— 对重要问题提出见解和处理办法
- `bulletin`（通报）—— 表彰先进、批评错误、传达重要情况
- `decision`（决定）—— 对重要事项作出决策和部署

### 步骤 4：生成公文初稿

按模板生成公文，遵循以下规范：
- 语言：简明扼要、庄重规范，使用公文常用语
- 结构：按文种规定的结构顺序组织
- 语气：上行文谦恭、下行文权威、平行文礼貌
- 用词：避免口语化、网络用语、口语表达
- 引用：引用公文须注明发文字号

### 步骤 5：格式校验

对用户提供的公文文本执行格式校验，检查清单：

**版头要素**：
- [ ] 发文机关标志（全称或规范化简称 + "文件"）
- [ ] 发文字号（机关代字 + 年份 + 序号，如"国发〔2026〕1号"）
- [ ] 签发人（上行文须标注）

**主体要素**：
- [ ] 标题（发文机关 + 事由 + 文种）
- [ ] 主送机关（顶格、全称、标点规范）
- [ ] 正文（结构层次序号规范）
- [ ] 附件说明（如有）
- [ ] 发文机关署名/印章
- [ ] 成文日期（用阿拉伯数字，年月日齐全）

**版记要素**：
- [ ] 抄送机关（如有）
- [ ] 印发机关和日期

**格式细节**：
- [ ] 层次序号规范：一、（一） 1. （1）
- [ ] 数字用法：统计数字用阿拉伯数字，概数用汉字
- [ ] 标点符号：标题中一般不用标点（书名号、引号除外）
- [ ] 引文标注：先引标题，后引发文字号

### 步骤 6：公文/政策信息提取

从政策文件中提取以下结构化信息：

```
文件标题：
发文机关：
发文字号：
发文日期：
主题分类：
核心内容摘要（200字以内）：
关键要点：
  1. ...
  2. ...
  3. ...
涉及对象/适用范围：
执行时限：
关联文件：
关键词标签：
```

### 步骤 7：政策对比分析

当用户提供多份政策文件时，输出结构化对比：

```
一、基本信息对比
  | 对比项 | 文件A | 文件B |
  |---|---|---|

二、核心条款对比
  | 条款主题 | 文件A 规定 | 文件B 规定 | 差异分析 |

三、新增/调整/废止条款
  - 新增：
  - 调整：
  - 废止：

四、影响分析
  - 对执行单位的影响：
  - 对管理对象的影响：
  - 衔接建议：
```

### 步骤 8：会议纪要生成

从会议记录中提炼结构化纪要：

```
会议名称：
会议时间：
会议地点：
主持人：
参会人员：
记录人：

一、会议议题
二、出席情况
三、讨论内容
  （一）议题一：...
    - 发言要点（发言人 + 核心观点）
    - 讨论结论
  （二）议题二：...
四、议定事项
  | 序号 | 事项内容 | 责任单位 | 完成时限 |
  |---|---|---|---|
五、其他事项
```

### 步骤 9：公众留言处理

对公众留言执行分类和答复生成：

**分类标准**：
| 类别 | 特征 | 紧急程度 |
|---|---|---|
| 咨询类 | 询问政策、流程、办事指南 | 一般 |
| 建议类 | 提出改进意见、建言献策 | 一般 |
| 投诉类 | 对服务或政策不满 | 较高 |
| 求助类 | 遇到困难需要协助 | 高 |
| 举报类 | 反映违法违规行为 | 紧急 |

**答复生成原则**：
1. 引用用户的具体诉求，不泛泛而谈
2. 引用相关政策文件条款作为依据
3. 给出明确的办理路径或答复结论
4. 署明答复单位和联系渠道
5. 态度诚恳，避免官话套话

## 已知限制

1. 公文起草生成的是初稿，实际发文前必须经人工审核和签发流程
2. 格式校验基于文本内容检查，不包含排版格式（字体、字号、页边距）的校验
3. 政策分析基于提供的文本内容，不包含政策执行效果的跟踪评估
4. 涉密公文处理不在本 Skill 范围内，请遵守保密规定
5. 会议纪要的质量取决于原始记录的完整程度

## 最佳实践

- 起草公文时，提供越具体的信息（发文字号、关联文件等），输出越精准
- 建议先用格式校验检查已有公文，再针对性修改
- 政策对比建议提供完整的政策原文，而非摘要
- 会议纪要建议原始记录包含发言人和时间戳，以提升准确性
- 公众留言答复需结合当地实际政策，建议人工审核后发布
